# Wallet Association Security v0.1

The transport is untrusted. Security comes from browser origin validation,
wallet approval, encrypted envelopes, session tokens, replay rejection, and
wallet-side signing policy.

## Algorithms

- Key agreement: X25519
- KDF: HKDF-SHA256
- AEAD: ChaCha20-Poly1305
- Envelope encoding: `base64(nonce || ciphertext || tag)`
- Nonce length: 12 bytes
- Symmetric key length: 32 bytes

## Handshake Key

The dapp and wallet derive the one-time handshake key from the X25519 shared
secret:

```text
salt = "native-wallet-association-v2:${origin}:${handshakeId}"
info = "handshake"
output = 32 bytes
```

## Session Key

RPC envelopes derive a session key from the raw session token bytes:

```text
salt = "native-wallet-association-v2:${origin}:${sessionId}"
info = "session"
output = 32 bytes
```

## Replay Policy

RPC payloads must include:

- `requestId`
- `issuedAt`
- `sessionTokenBase64`

Wallets must reject:

- requests older than 60 seconds
- requests issued more than 60 seconds in the future
- duplicate `requestId` values for the same session
- missing, expired, or mismatched session tokens

Wallet replay caches must be bounded. If a wallet evicts old request IDs, it
must evict them in deterministic insertion order.

## Session Tokens

Session tokens are random 32-byte values generated by the wallet. Dapps store
the token to resume a session. Wallets and dapps must reject tokens that do not
decode to exactly 32 bytes. Wallets store token material in secure local storage
when available and must revoke tokens when the user revokes a session.

Browser dapp implementations may persist tokens in `localStorage` for v0.1.
This enables session resume but means same-origin XSS can steal the session
token. Dapps with stronger requirements should provide a custom storage layer.

## Origin Binding

Browser transports must provide an HTTP `Origin` header. Wallets must bind
sessions to that origin and reject RPC requests whose browser origin does not
match the session origin.

For localhost v0.1, a valid browser origin is exactly an `http` or `https`
origin with scheme, host, and optional port only. Wallets must reject `null`,
file origins, credentials, paths, queries, and fragments.

## Signing Policy

The wallet decides whether a session can sign silently or must prompt for each
signature. The default policy is `prompt`.

## Threat Model

v0.1 considers:

- localhost probing and fingerprinting
- malicious same-machine processes
- stolen browser storage session tokens
- replayed RPC envelopes
- wallet impersonation by a local service
- origin spoofing attempts
- denial of service through oversized requests or handshake flooding

v0.1 does not require trusting localhost. Localhost is only a delivery path.
